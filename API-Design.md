If the API is generated by auto coding configuration, this design process can be ignored.

The documentation of the API design can be presented in one of the following ways:
- Swagger
- Word
- Markdown

All APIs should comply with RESTful API design principles. This includes public APIs that are exposed to customers and internal APIs that are only used for internal service-to-service calls.

# Design Goals

- Make APIs easy to understand and predictable.
- Keep the same patterns across products and services.
- Clearly define every request, response, validation rule, and error case.
- Prefer standard HTTP semantics over custom conventions.

# General Specifications

- Usually a backend service uses a separate path as the routing rule for the API gateway. For example, we use `/kb` to route all requests to the `kbapi` service.
- For general RESTful design concepts, refer to [REST API Tutorial](https://restfulapi.net/).
- The input and output parameters of each API must be clearly defined in the design document.
- All results should be returned in JSON format unless there is a special and justified reason.
- Every API design document should include:
  - Endpoint and HTTP method
  - Parameter definitions
  - Request body definition
  - Response body definition
  - Success status codes
  - Error status codes
  - Example requests and responses

# Route Design

## 1. Use Service Prefixes

Use a stable service prefix at the beginning of the route.

- Good:
  - `/kb/articles`
  - `/integration/webhooks`
  - `/global/guidelines`
  - `/aikhub/{productId}/knowledgeSources`

## 2. Use Nouns for Resources

Routes should represent resources, not verbs.

- Good:
  - `GET /kb/articles`
  - `GET /kb/articles/{articleId}`
  - `POST /kb/articles`
- Avoid:
  - `GET /kb/getArticles`
  - `POST /kb/createArticle`
  - `POST /kb/deleteArticle`

## 3. Use Plural Names for Collections

Prefer plural names for collection resources.

- Good: `/articles`, `/webhooks`, `/knowledgeSources`
- Avoid: `/article`, `/webhook`, `/knowledgeSource`

## 4. Use Path Parameters for Resource Identity

Use path parameters for the unique identity of a resource.

- Good:
  - `GET /integration/webhooks/{id}`
  - `PUT /global/guidelines/{guidelineId}`

## 5. Use Nested Routes Only When the Ownership Is Clear

Nested routes are appropriate when a child resource clearly belongs to a parent resource.

- Good:
  - `/aikhub/{productId}/knowledgeSources`
  - `/kb/articles/{articleId}/revisions`

Avoid deep nesting when the relationship is weak or when the route becomes difficult to read.

## 6. Use Action Suffixes for Non-CRUD Operations

When an operation is not standard CRUD, use a command-style suffix after the resource.

- Good:
  - `POST /integration/webhooks/{id}:resetSecretKey`
  - `POST /aikhub/knowledgeSources/{id}:retry`
  - `POST /reviews/{id}:submit`

Do not create separate verb-style top-level routes such as `/resetWebhookSecretKey`.

# HTTP Method Guidelines

- `GET`: Read resource data. Must not create or modify server state.
- `POST`: Create a resource or perform a non-idempotent action.
- `PUT`: Replace the whole resource or perform a full update.
- `DELETE`: Remove a resource.
- Keep the behavior consistent within the same service.

# Status Code Guidelines

- `200 OK`: Successful read, update, or action.
- `201 Created`: Resource created successfully.
- `204 No Content`: Resource deleted successfully, or successful operation with no response body.
- `400 Bad Request`: Invalid input, invalid query values, or validation failure.
- `401 Unauthorized`: Authentication is missing or invalid.
- `403 Forbidden`: The caller is authenticated but does not have permission.
- `404 Not Found`: Resource does not exist.
- `409 Conflict`: Request conflicts with the current state of the system.
- `500 Internal Server Error`: Unexpected server-side failure.

# Request Design

## 1. Parameter Placement

- Use path parameters for resource identity.
- Use query parameters for filtering, pagination, sorting, and lightweight options.
- Use request body for create and update payloads.

## 2. Common Query Parameters

Common query parameters used in Comm100 APIs include:

- `siteId`
- `keywords`
- `include`
- `sortBy`
- `sortOrder`
- `pageIndex`
- `pageSize`

Document all allowed values and defaults explicitly.

## 3. Validation Rules

Validation rules should be written in the API design document, including:

- Required vs optional fields
- Allowed enum values
- String length limits
- Numeric ranges
- Date/time format
- Whether a field can be `null`

## 4. Naming Conventions

- Prefer `camelCase` for JSON fields, query parameters, and request body fields.
- Use clear, domain-specific names.
- Avoid abbreviations unless they are already standard in the domain.

- Good:
  - `createdTime`
  - `lastUpdatedTime`
  - `sourceType`
  - `pageIndex`
- Avoid:
  - `crtTm`
  - `upd`
  - `pIdx`

# Response Design

## 1. JSON Response Format

All APIs should return JSON.

- Single-resource endpoints should return a single object.
- List endpoints should return a list payload or a list envelope with pagination metadata.

## 2. Data Type Conventions

- Use `guid` for globally unique resource identifiers when applicable.
- Use `int` for numeric identifiers such as `siteId` where appropriate.
- Use ISO 8601 UTC format for date/time fields, for example: `2026-03-11T10:30:00Z`.
- Use `bool` for true/false flags.

## 3. Include Metadata Only When Needed

If some fields are expensive or optional, expose them through an `include` query parameter.

Example:

- `GET /aikhub/knowledgeSources?include=metadata`

# Pagination Guideline

Prefer page-based pagination unless there is a strong reason to use another style.

Recommended query parameters:

- `pageIndex`: starts from `1`
- `pageSize`: set a default value and a maximum value

Recommended response envelope:

```json
{
  "items": [
    {
      "id": "4fc10466-3900-4d0d-89d8-635afb8a2c78",
      "name": "Sample Item"
    }
  ],
  "pageIndex": 1,
  "pageSize": 20,
  "total": 135,
  "nextPage": "/service/items?pageIndex=2&pageSize=20",
  "previousPage": null
}
```

If a service already uses a different pagination envelope, keep it consistent within that service and document it clearly.

# Error Response Guideline

Use a small and consistent error body. At minimum, return:

- `error`: machine-readable error code or short error text
- `message`: human-readable message
- `details`: optional structured details for debugging or validation feedback

Example:

```json
{
  "error": "InvalidQueryParameter",
  "message": "The value of pageSize must be between 1 and 100.",
  "details": {
    "field": "pageSize",
    "allowedRange": "1-100"
  }
}
```

Recommended principles:

- Keep `error` stable for programmatic handling.
- Keep `message` clear for people.
- Do not expose sensitive internal details in production responses.

- Description: Create a new webhook.

| - | - | - | - |

# Review Checklist

Before finalizing an API design, verify the following:

- The route is resource-oriented and easy to understand.
- The HTTP method matches the behavior.
- Parameters are placed in the correct location.
- Required fields, defaults, and allowed values are documented.
- Response body fields are clearly defined.
- Success and error status codes are documented.
- Example requests and responses are included.
- Naming follows the agreed conventions.
- Authentication and authorization are documented.
- Pagination, sorting, and filtering rules are clear.
